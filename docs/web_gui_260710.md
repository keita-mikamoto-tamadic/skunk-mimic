# Web GUI 操作手順（motor monitor / controller）

FOCTIVE 単軸を **PC のブラウザ**から表示・操作するための手順。2026-07-10 時点。

- **web_monitor**（表示）… `tools/gui/web_monitor.py`。dataflow 非依存。`dora topic echo`
  で `motor_status` を拾い、デコードして HTTP/SSE でブラウザへ（uPlot グラフ付き）。
- **web_controller**（操作）… `src/python/web_controller/web_controller.py`。dataflow に
  配線した本物の dora ノードで、`motor_commands`(AxisRef) を送る。PC 側 daemon に deploy。

構成例: robot = `10.42.0.235`（coordinator＋既定 daemon, can0, リポジトリ
`/home/minijet/skunk-mimic`）、PC = `10.42.0.1`、PC daemon の machine-id = `pc`。

---

## 0. 前提

- robot 側: `dora up`（coordinator＋既定 daemon）稼働、`bash can_setup.bash` で can0 UP。
- **閉ループ制御（電流/速度/位置/pd/cpos/cvel）は事前に電気角キャリブが必要。**
  端末版 `foctive_controller` の `setting calib <volt_d>` で済ませておく。
  電圧制御 `v` はキャリブ不要。

---

## 1. 表示だけ（web_monitor, dataflow 非依存）

監視したい dataflow（foctive でも mimic でも）が動いていれば、それを自動発見して映す。
前提は dataflow 側の `_unstable_debug.enable_debug_inspection: true` のみ。

```bash
export DORA_COORDINATOR_ADDR=10.42.0.235      # robot の coordinator
cd tools/gui
ROBOT_CONFIG=robot_config/foctive_motor_test.json python3 web_monitor.py
#   ROBOT_CONFIG は軸名/CAN ID 表示用（任意。未指定なら #0,#1,...）
```

ブラウザ: `http://localhost:8765/`（PC で動かした場合）。
依存なし（stdlib のみ、uv 不要、`dora` CLI だけ要る）。グラフ操作は
`tools/gui/README.md` 参照（信号選択・時間窓 5/10/30/60s・Pause でズーム）。

---

## 2. 操作（web_controller, 分散構成）

操作ノードを PC 側 daemon に deploy し、ブラウザから `motor_commands` を送る。
descriptor は `dataflow_foctive_web_control.yaml`。

```bash
# ① PC: machine-id 付き daemon を robot coordinator に参加（別ターミナル・常駐）
dora daemon --coordinator-addr 10.42.0.235 --machine-id pc
#   machine-id は descriptor の _unstable_deploy.machine（= pc）と一致させる

# ② dataflow を起動（robot のリポジトリルートで叩くのが確実）
export DORA_COORDINATOR_ADDR=10.42.0.235
dora start dataflow_foctive_web_control.yaml

# ③ PC: 操作ノード本体を起動（path: dynamic → localhost の pc daemon にアタッチ）
cd src/python && uv run web_controller/web_controller.py

# ④ PC ブラウザ
#   http://localhost:8770/
```

### ブラウザ操作

- **■ SERVO OFF**（赤・最上部）… いつでも停止。
- **電圧 VOLTAGE**（volt_d, volt_q, vir_ang_freq）… キャリブ不要。まずこれで小さく通す。
- **電流 CURRENT**（cur_d, cur_q）
- **速度 VELOCITY / impedance**（vel, kp, kd, accel_limit）… 既定 kp=1,kd=1,accel=0
- **位置 POSITION / impedance**（pos, kp, kd, accel_limit）
- **インピーダンス POSITION_PD**（pos, vel, torq(FF), kp_scale, kd_scale）
- **カスケードPID 位置 cpos**（pos, accel_limit）… FOCTIVE 専用
- **カスケードPID 速度 cvel**（vel, accel_limit）… FOCTIVE 専用

各行の Send で送信（自動送信なし）。送信内容は "last sent" に表示。
指令マッピングは更新後の `foctive_controller` と一致（MotorState 2/3/6/7/8/9/10）。

### 表示と併用

web_monitor（§1）は別プロセス・dataflow 非依存なので、この dataflow でもそのまま並行起動可。
→ **操作(8770)** と **表示(8765)** を両方 PC ブラウザで。

## 3. 停止

```bash
dora list                     # dataflow id 確認
dora stop <dataflow-id>
# web_controller.py / web_monitor.py は Ctrl-C。PC daemon も不要なら Ctrl-C。
```

## 注意・ハマりどころ

- **実モータが動く。** SERVO OFF を手元に。最初の通しは電圧で小さく。
- **robot 側 config は絶対パスにする。** `_unstable_deploy` 入り dataflow の
  ノードは `<daemon cwd>/_work/<session-id>`（空ディレクトリ）で実行されるため、
  相対パスだと config を開けない。descriptor では
  `ROBOT_CONFIG: /home/minijet/skunk-mimic/robot_config/foctive_motor_test.json`
  のように robot 上の絶対パスを指定している。
- **dora CLI が coordinator を向いているか。** `export DORA_COORDINATOR_ADDR=<robot-ip>`
  を忘れると `127.0.0.1:6013` に繋ぎにいって "Connection refused"。
- **`dora node add` は使わない。** rc1 では deploy を無視して robot daemon に置くため、
  PC 配置は start 時 deploy（この descriptor 方式）でのみ可能。
- **分散の通し（PC daemon＋dynamic アタッチ＋cross-daemon 到達）は実機テストで確認のこと。**
